# 二开分支与 Fork 同步流程说明

## 1. 文档目的

这份文档用于说明本仓库在 GitHub Fork 模式下的分支关系、职责边界和标准更新流程，避免后续维护人员直接在错误分支上开发或把官方更新错误合入二开分支。

核心原则只有一句话：

- `main` 负责跟官方开源版本同步。
- `nuohe369-main` 负责承载二开功能。
- 官方更新先在缓冲分支验证，再决定是否回合到 `nuohe369-main`。

## 2. 远端关系

当前仓库有两个关键远端：

- `upstream`：官方开源仓库 `QuantumNous/new-api`
- `origin`：自己的 Fork 仓库 `nuohe369/new-api`

职责说明：

- `upstream` 用来获取官方最新更新。
- `origin` 用来保存自己的分支和二开成果。

## 3. 分支职责

### 3.1 `main`

`main` 是 Fork 出来的官方同步分支，不承担二开日常开发任务。

要求：

- 尽量保持和 `upstream/main` 接近。
- 只用于承接官方开源版更新。
- 不在这个分支上直接做业务二开。

### 3.2 `nuohe369-main`

`nuohe369-main` 是当前项目的二开主分支。

要求：

- 所有业务定制功能都应该进入这个分支。
- 日常开发、修复、页面改动、后台功能改动都以这个分支为主。
- 不直接把官方更新硬合进来，必须先过缓冲分支。

### 3.3 `buffer/*`

`buffer/*` 是缓冲分支，用来做“官方更新合入二开分支”的预演。

建议命名：

- `buffer/merge-main-into-nuohe369-main-YYYYMMDD`

职责：

- 从 `nuohe369-main` 拉出。
- 把最新 `main` 合并进来。
- 处理冲突、做验证、给负责人确认。
- 确认通过后，才允许把缓冲分支回合到 `nuohe369-main`。

## 4. 绝对不要做的事

- 不要直接在 `main` 上做二开功能。
- 不要把 `upstream/main` 直接合到 `nuohe369-main`，跳过缓冲分支。
- 不要在工作区有未提交改动时直接做大范围同步。
- 不要把 `release/` 打包产物提交进仓库。
- 不要把 `归档.zip` 提交进仓库。

## 5. 标准更新流程

每次要吸收官方更新时，必须按下面顺序执行。

### 步骤 1：先处理当前二开中的未提交改动

如果 `nuohe369-main` 上还有本地未提交功能，必须先处理干净：

- 能提交的先提交。
- 暂时不准备提交的先暂存。
- 不允许带着脏工作区直接进入同步流程。

原因：

- 后面要切换分支、拉缓冲分支、做合并。
- 如果本地还有散落改动，后续很难判断哪些是旧功能、哪些是合并冲突。

### 步骤 2：先同步 `main`

目标是让 `main` 代表“当前官方最新基线”。

标准动作：

1. 拉取 `upstream` 最新代码。
2. 切到本地 `main`。
3. 把 `main` 更新到 `upstream/main`。
4. 视情况推送到 `origin/main`。

这样做完以后，`main` 的意义就是“官方开源版当前最新状态”。

### 步骤 3：从 `nuohe369-main` 拉出缓冲分支

缓冲分支必须从二开分支拉出，而不是从 `main` 拉出。

原因：

- 目标是验证“官方更新进入二开版本后会不会冲掉我们的功能”。
- 所以缓冲分支天然应该以二开版本为底。

### 步骤 4：把最新 `main` 合进缓冲分支

在缓冲分支中执行：

- 把同步后的 `main` 合并进来。

这样得到的是：

- 官方最新更新
- 加上当前二开功能

但这一步还只是试合并，不代表正式完成。

### 步骤 5：处理冲突并验证

冲突处理时优先保证二开业务不被破坏，重点检查：

- 邀请返利
- 邀请有礼页面
- 钱包管理
- 兑换码参与返利
- 后台支付开关
- 管理员自定义邀请码

同时检查官方更新是否引入了对这些模块的结构性影响。

### 步骤 6：提交给负责人确认

缓冲分支完成后，不直接合回 `nuohe369-main`。

必须先输出：

- 官方这次更新了什么
- 哪些文件发生冲突
- 哪些二开逻辑做了兼容处理
- 目前有哪些风险点

负责人确认后，才能进入下一步。

### 步骤 7：确认后回合到 `nuohe369-main`

确认通过后再把缓冲分支合并回 `nuohe369-main`，然后推送到 `origin/nuohe369-main`。

## 6. 冲突处理优先级

遇到冲突时，按下面顺序判断：

1. 先看官方更新是不是安全修复、底层兼容修复或关键逻辑修复。
2. 再看二开功能是否依赖被改动的结构。
3. 优先保住二开业务结果，但不要把官方安全修复直接丢掉。
4. 如果官方和二开都改了同一个模块，要做兼容，不要简单覆盖其中一方。

## 7. 当前二开功能范围

截至当前维护阶段，二开中已经明确存在或正在处理的功能包括：

- 钱包管理独立页面化
- 邀请有礼独立菜单和页面
- 邀请用户列表与返现收益记录优化
- 表格结构与后台其他页面风格统一
- 手机端适配
- 兑换码可配置参与返利
- 后台可配置关闭支付功能
- 钱包页面在关闭支付后隐藏充值相关区块，但保留账户统计和兑换码充值
- 管理员自定义邀请码

后续同步官方更新时，上述功能都属于重点回归范围。

## 8. 每次同步前的检查清单

- 当前是否在正确分支
- 工作区是否干净
- 本地重要功能是否已经提交
- `release/` 是否被排除
- `归档.zip` 是否被排除
- `main` 是否已经同步到 `upstream/main`
- 是否已经从 `nuohe369-main` 创建缓冲分支

## 9. 每次同步后的检查清单

- 缓冲分支是否已成功合入最新 `main`
- 关键页面是否能打开
- 用户、新增编辑弹窗是否正常
- 邀请返利相关接口是否正常
- 支付开关逻辑是否仍生效
- 兑换码返利逻辑是否仍生效
- 是否已经形成给负责人确认的结果说明

## 10. 给下一位维护者的结论

如果你是下一位接手的人，请记住：

- `main` 不是二开主战场。
- `nuohe369-main` 才是实际业务主分支。
- 同步官方更新时，不要直接碰正式二开分支。
- 先提交本地改动，再同步 `main`，再建 `buffer/*`，再试合并，确认后才正式回合。
