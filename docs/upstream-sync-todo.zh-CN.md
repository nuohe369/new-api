# 官方更新吸收任务 Todo

## 1. 任务目标

本次任务的目标不是直接修改功能代码，而是按既定分支策略，把官方开源版的最新更新安全吸收到二开流程中。

最终目标：

- `main` 继续作为官方同步分支
- `nuohe369-main` 继续作为二开主分支
- 官方更新先在缓冲分支中验证
- 确认兼容后再回合到 `nuohe369-main`

## 2. 当前已知状态

截至 2026-04-19，当前仓库状态如下：

- 当前工作分支：`nuohe369-main`
- `main` 对应的是 Fork 后的官方同步分支
- `nuohe369-main` 对应的是二开主分支
- `upstream/main` 已经领先 `origin/main`
- 二开中还有“管理员自定义邀请码”功能未正式提交

当前待提交文件：

- `controller/user.go`
- `model/user.go`
- `web/src/components/table/users/modals/AddUserModal.jsx`
- `web/src/components/table/users/modals/EditUserModal.jsx`

当前明确不应进入版本库的内容：

- `release/`
- `归档.zip`

## 3. 执行顺序

下面顺序不要调整。

### Todo 1：先提交管理员自定义邀请码功能

目标：

- 先把当前 `nuohe369-main` 上未提交的“管理员自定义邀请码”功能收口成单独提交。

要求：

- 只提交相关 4 个文件。
- 不要混入 `release/`。
- 不要混入 `归档.zip`。

完成标准：

- `git status` 中不再出现这 4 个文件的未提交修改。

### Todo 2：同步 `main` 到官方最新

目标：

- 让本地和远端 `main` 重新代表“官方最新基线”。

要求：

- 从 `upstream/main` 获取最新更新。
- 更新本地 `main`。
- 视情况推送到 `origin/main`。

完成标准：

- `main` 与 `upstream/main` 对齐。

### Todo 3：从 `nuohe369-main` 拉出缓冲分支

建议分支名：

- `buffer/merge-main-into-nuohe369-main-20260419`

目标：

- 不直接改正式二开分支。
- 先在缓冲分支做官方更新吸收验证。

完成标准：

- 缓冲分支创建完成，并以 `nuohe369-main` 为基线。

### Todo 4：把最新 `main` 合进缓冲分支

目标：

- 在缓冲分支中预演“官方更新进入二开版本”的结果。

要求：

- 所有冲突都只在缓冲分支上处理。
- 不要提前动 `nuohe369-main`。

完成标准：

- 缓冲分支可以正常完成合并。

### Todo 5：整理官方更新内容

目标：

- 让下一位处理者和负责人清楚本次官方到底改了什么。

建议整理内容：

- 安全修复
- 支付相关改动
- 日志与审计改动
- 模型或渠道兼容性修复
- Docker / 部署层更新

完成标准：

- 形成一份简明更新说明，能说明“本次为什么值得吸收”。

### Todo 6：处理冲突并保住二开能力

重点检查模块：

- 钱包管理
- 邀请有礼
- 邀请返利
- 兑换码参与返利
- 后台支付开关
- 用户新增/编辑弹窗
- 管理员自定义邀请码

完成标准：

- 缓冲分支上的冲突全部解决。
- 二开关键逻辑没有明显被官方更新覆盖掉。

### Todo 7：做回归验证

至少验证下面这些点：

- 邀请有礼页面可正常打开
- 邀请用户列表可正常展示
- 返现收益记录可正常展示
- 钱包管理页面可正常打开
- 支付开关打开后，钱包充值相关区块隐藏
- 账户统计模块仍保留
- 兑换码充值仍保留
- 兑换码可设置参与返利
- 管理员可新增和编辑自定义邀请码

完成标准：

- 关键路径无明显阻塞问题。

### Todo 8：输出确认材料

提交确认前，需要整理：

- 本次合并涉及的主要冲突点
- 二开保留了哪些功能
- 官方更新吸收了哪些内容
- 还剩哪些风险未验证

完成标准：

- 负责人看到后可以直接判断是否允许正式回合。

### Todo 9：负责人确认后再回合到 `nuohe369-main`

要求：

- 未确认前，不要直接把缓冲分支回合到正式二开分支。

完成标准：

- 收到明确确认后，再进行正式合并。

## 4. 风险提醒

- 如果跳过 Todo 1，后续很容易把“管理员自定义邀请码”功能混进同步冲突里，导致很难回溯。
- 如果直接把官方更新合到 `nuohe369-main`，一旦冲突处理失误，会直接污染正式二开分支。
- 如果把 `release/` 或 `归档.zip` 混入提交，会让版本历史变脏，后续同步和回滚都麻烦。

## 5. 最终交付标准

本轮任务真正完成时，应满足：

- `main` 已同步官方最新
- `nuohe369-main` 仍保持为二开主分支
- 官方更新已在缓冲分支完成兼容验证
- 负责人已确认是否允许正式回合
- 不需要入库的产物没有被提交
