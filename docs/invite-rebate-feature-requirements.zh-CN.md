# 邀请返现功能二开需求文档

## 1. 文档目标

在尽量兼容原版数据库结构的前提下，对当前邀请奖励功能进行二开，修正“用户一注册就给邀请人返现”的错误业务逻辑，改为：

1. 后台可配置“被邀请用户注册后赠送多少余额”。
2. 后台可配置“下级用户成功预存后的返现百分比”。
3. 用户端可查看邀请收益统计与已邀请用户列表。
4. 数据库层面只允许新增表，或在原有表上新增字段，不允许破坏原有字段和原版兼容性。

## 2. 现状分析

### 2.1 现有数据结构

当前邀请体系主要依赖以下结构：

- `users` 表已有字段：
  - `aff_code`：用户自己的邀请码
  - `aff_count`：邀请人数
  - `aff_quota`：待划转邀请收益
  - `aff_history`：累计邀请收益，对应结构体字段 `AffHistoryQuota`
  - `inviter_id`：邀请人用户 ID
- `options` 表已有配置项：
  - `QuotaForInviter`
  - `QuotaForInvitee`
- `top_ups` 表保存充值订单，但当前没有任何邀请返现结算字段。

对应代码位置：

- `model/user.go`
- `model/topup.go`
- `model/option.go`

### 2.2 当前错误行为

当前邀请奖励在“注册成功”阶段直接结算，而不是在“被邀请用户成功充值/预存”后结算。

现有逻辑位置：

- `model/user.go`
  - `Insert(inviterId int)`
  - `FinalizeOAuthUserCreation(inviterId int)`
  - `inviteUser(inviterId int)`

当前行为为：

1. 被邀请用户注册成功后，如果配置了 `QuotaForInvitee`，系统立即给被邀请用户赠送余额。
2. 同一时刻，如果配置了 `QuotaForInviter`，系统立即给邀请人累计 `aff_quota` 和 `aff_history_quota`。
3. 邀请人的收益不是由“下级成功预存”触发，而是由“注册成功”触发。

这与目标业务不一致。

### 2.3 当前充值链路未参与邀请返现

当前充值成功逻辑分散在以下位置：

- `controller/topup.go`
  - 易支付回调成功后，仅给充值用户自己加余额
- `model/topup.go`
  - `Recharge`
  - `RechargeCreem`
  - `RechargeWaffo`
  - `ManualCompleteTopUp`

这些逻辑都只处理“给充值用户本人加额度”，没有处理“给上级返现”。

### 2.4 当前前端能力不足

当前前端仅展示邀请汇总卡片，没有邀请用户列表和返现明细能力。

对应位置：

- `web/src/components/topup/InvitationCard.jsx`
- `web/src/components/topup/index.jsx`
- `web/src/pages/Setting/Operation/SettingsCreditLimit.jsx`

现状表现：

1. 管理后台只有两个输入项：
  - `QuotaForInviter`
  - `QuotaForInvitee`
2. 用户端只显示：
  - 邀请人数
  - 待使用收益
  - 总收益
3. 没有“已邀请用户列表”接口和页面。
4. 没有结构化的返现收益记录。

## 3. 核心问题

### 3.1 业务问题

- 返现触发时机错误：注册即返现。
- 无法配置“下级预存返现百分比”。
- 无法追踪每一笔返现来源。
- 用户无法查看自己邀请了哪些人。

### 3.2 技术问题

- 目前只有聚合字段，没有结构化返现流水。
- 充值链路分散，多支付方式下需要统一返现结算入口。
- 现有 `QuotaForInviter` 是“固定额度”，无法表达“百分比返现”。
- 历史 `aff_quota/aff_history_quota` 已混入旧逻辑数据，上线前需要明确是否做数据修正。

## 4. 二开目标

本次二开目标定义如下：

1. 邀请关系仍在注册时建立。
2. 被邀请用户注册成功后，可按后台配置赠送注册余额。
3. 邀请人不再因为“下级注册成功”立即获得返现。
4. 邀请人仅在“下级成功预存/充值到账”后，按后台配置比例获得返现收益。
5. 用户端可以查看：
  - 累计返现收益
  - 待划转返现收益
  - 已邀请用户列表
  - 建议同时支持返现明细列表

## 5. 需求范围

### 5.1 本期范围

- 邀请注册赠送余额
- 下级预存返现比例
- 邀请收益结算
- 邀请用户列表
- 返现收益明细
- 管理后台配置项改造
- 数据库兼容性迁移

### 5.2 本期不包含

- 多级分销
- 邀请关系变更/解绑
- 已退款订单的自动反向冲销
- 订阅类订单是否参与返现

说明：

“订阅类订单是否参与返现”建议本期默认不纳入。当前文档将“预存返现”定义为普通余额充值/充值到账场景；若后续业务要求订阅也参与，应单独补充规则。

## 6. 详细业务需求

### 6.1 邀请关系建立

1. 用户注册时填写邀请码，系统通过 `aff_code` 找到邀请人，并写入新用户 `inviter_id`。
2. 普通注册与 OAuth 注册保持一致逻辑。
3. 一个用户只能绑定一个邀请人，注册完成后不可修改。

### 6.2 被邀请用户注册赠送

1. 后台可配置“邀请注册赠送余额”。
2. 当新用户使用有效邀请码完成注册时，系统给该新用户赠送对应余额。
3. 该配置建议复用现有 `QuotaForInvitee`，避免新增表字段，兼容原版 `options` 表。

### 6.3 邀请人返现规则

1. 邀请人不再因下级注册成功而获得固定奖励。
2. 当下级用户发生成功充值/预存到账时，系统按返现比例给邀请人结算收益。
3. 返现计算基数定义为“该笔订单实际到账的额度值”，不是支付金额字符串，也不是前端原始输入值。
4. 返现金额计算公式：

```text
返现额度 = floor(实际到账额度 * 返现比例)
```

5. 返现先进入邀请收益池，即累计到现有 `aff_quota`，并同步累计到 `aff_history_quota`。
6. 用户继续使用现有“邀请额度划转到余额”能力，不改变既有交互习惯。

说明：

按“实际到账额度”结算，而不是按 `Money` 结算，原因是当前不同支付方式对 `Amount` / `Money` 字段语义不完全一致，而系统最终给用户增加的资产是 `quota`。

### 6.4 返现触发时机

以下场景需要触发返现结算：

- 易支付充值成功
- Stripe 充值成功
- Creem 充值成功
- Waffo 充值成功
- 管理员补单成功

返现触发前置条件：

1. 订单状态从 `pending` 成功切换为 `success`。
2. 充值用户存在有效 `inviter_id`。
3. 当前订单未做过邀请返现结算。

### 6.5 幂等要求

支付回调可能重复到达，因此返现结算必须具备幂等性。

要求：

1. 同一笔充值订单最多生成一条返现结算记录。
2. 同一笔订单重复回调，不得重复给邀请人加收益。
3. 补单接口也必须走同样的幂等校验。

### 6.6 用户端展示

用户端至少要提供以下信息：

#### 收益统计

- 邀请人数
- 待划转收益
- 累计收益

#### 已邀请用户列表

建议字段：

- 被邀请用户 ID
- 用户名或脱敏邮箱
- 注册时间
- 是否已发生充值
- 累计充值额度
- 累计返现额度
- 最近一次充值时间

#### 返现收益明细

建议字段：

- 时间
- 下级用户
- 来源订单号
- 该笔到账额度
- 返现比例
- 本次返现额度
- 状态

如果产品只要求“收益统计 + 邀请用户列表”，明细可以做成二期；但从财务可追溯性和客服排查角度，建议本期一并实现。

## 7. 数据库改造方案

## 7.1 兼容原则

本项目要求同时兼容 SQLite、MySQL、PostgreSQL，因此数据库设计必须遵守以下原则：

1. 不删除原有字段。
2. 不修改原有字段语义到无法兼容原版。
3. 优先新增表或新增字段。
4. 避免使用 JSONB、数组、数据库特定函数。
5. 金额/比例优先使用整数存储，避免跨数据库精度差异。

### 7.2 复用现有字段

保留并继续使用 `users` 表字段：

- `inviter_id`：邀请关系
- `aff_count`：邀请人数
- `aff_quota`：待划转收益
- `aff_history`：累计收益

字段语义调整为：

- `aff_count`：已成功邀请注册的下级数量
- `aff_quota`：待划转邀请返现余额
- `aff_history`：累计邀请返现收益

### 7.3 配置项改造

推荐方案：

1. 复用 `QuotaForInvitee`
   - 含义改为“被邀请用户注册赠送余额”
2. 废弃 `QuotaForInviter`
   - 不再参与结算逻辑
   - 可在后台标记为旧字段，或直接隐藏
3. 新增 `options` 配置项：
   - `InviteTopupRebateRate`

建议存储方式：

- 使用整数，单位为基点 `bp`
- 例如：
  - `500` = 5%
  - `1000` = 10%
  - `2500` = 25%

这样可以避免浮点精度问题，并保持三库兼容。

### 7.4 新增返现流水表

建议新增表：`invite_rebate_records`

建议字段如下：

| 字段 | 类型建议 | 说明 |
| --- | --- | --- |
| `id` | int | 主键 |
| `inviter_user_id` | int | 邀请人 ID |
| `invitee_user_id` | int | 下级用户 ID |
| `topup_id` | int | 来源充值记录 ID，可为空 |
| `trade_no` | varchar(255) | 来源订单号，唯一索引 |
| `payment_method` | varchar(50) | 支付方式 |
| `credited_quota` | int | 该笔订单实际到账额度 |
| `rebate_rate_bp` | int | 返现比例，单位 bp |
| `rebate_quota` | int | 本次返现额度 |
| `status` | int | 1=已结算，2=已冲销，默认 1 |
| `created_at` | bigint | 创建时间 |
| `updated_at` | bigint | 更新时间 |
| `remark` | varchar(255) | 备注，可选 |

索引建议：

- `trade_no` 唯一索引
- `inviter_user_id` 普通索引
- `invitee_user_id` 普通索引
- `created_at` 普通索引

说明：

1. 有了这张表，就可以可靠支撑“返现明细”与“幂等校验”。
2. 邀请用户列表仍可基于 `users.inviter_id` 直接查。
3. 累计返现可从该表聚合，也可继续同步写入 `users.aff_quota / aff_history` 作为汇总缓存。

### 7.5 是否需要新增邀请关系表

本期不建议新增独立邀请关系表，原因如下：

1. 现有 `users.inviter_id` 已经能表达一对一邀请关系。
2. 邀请用户列表可以直接通过 `users WHERE inviter_id = ?` 获取。
3. 最核心缺口不是“关系”，而是“返现流水”。

因此本期最小闭环方案为：

- 保留 `users.inviter_id`
- 新增 `invite_rebate_records`

## 8. 后端改造要求

### 8.1 注册链路改造

调整目标：

1. 保留被邀请用户注册赠送。
2. 取消邀请人注册即返现。

需要修改的核心位置：

- `model/user.go`
  - `Insert`
  - `FinalizeOAuthUserCreation`
  - `inviteUser`

建议改法：

1. 保留 `inviter_id` 写入。
2. 保留 `QuotaForInvitee` 给新用户赠送。
3. 删除或停用注册阶段对邀请人的 `aff_quota/aff_history_quota` 累加。
4. `aff_count` 仍在注册成功后累加，表示有效邀请注册人数。

### 8.2 充值成功统一结算

新增统一服务方法，建议放在 `model` 或 `service` 层，例如：

```go
SettleInviteRebateForTopup(tx *gorm.DB, topUp *TopUp, creditedQuota int) error
```

职责：

1. 根据 `topUp.UserId` 找到充值用户。
2. 检查是否存在 `inviter_id`。
3. 检查订单是否已经结算过返现。
4. 计算返现额度。
5. 写入 `invite_rebate_records`。
6. 原子更新邀请人的 `aff_quota` 和 `aff_history_quota`。

该方法应被以下流程复用：

- 易支付回调成功
- Stripe 成功
- Creem 成功
- Waffo 成功
- 管理员补单成功

### 8.3 查询接口

建议新增以下接口：

#### 用户侧

- `GET /api/user/invite/summary`
  - 返回邀请码、邀请人数、待划转收益、累计收益
- `GET /api/user/invite/users`
  - 分页返回已邀请用户列表
- `GET /api/user/invite/rebates`
  - 分页返回返现明细列表

也可以在不破坏现有接口的前提下，直接扩展 `GET /api/user/self` 返回字段，但列表类数据仍建议独立接口。

#### 管理侧

后台配置项保存继续走现有 `option` 机制即可，无需新增独立配置表。

## 9. 前端改造要求

### 9.1 后台设置页面

当前 `SettingsCreditLimit.jsx` 中的两个邀请相关配置需要调整为：

1. `被邀请用户注册赠送余额`
   - 复用 `QuotaForInvitee`
2. `下级预存返现比例`
   - 绑定新配置 `InviteTopupRebateRate`
   - 输入和展示建议为百分比

`QuotaForInviter` 不应继续以“邀请新用户奖励额度”的形式暴露给管理员。

### 9.2 用户端邀请页面

在当前邀请卡片基础上增加：

1. 邀请收益统计区
2. 已邀请用户列表
3. 返现收益明细列表

建议放置位置：

- 继续放在 `web/src/components/topup/` 下，保持现有入口不变

## 10. 历史数据迁移建议

这是本需求最容易遗漏但必须提前确认的部分。

### 10.1 现有历史数据问题

当前线上如果已经运行旧逻辑，则以下数据已经被污染：

- `users.aff_quota`
- `users.aff_history`
- 邀请相关系统日志

这些历史值包含“下级仅注册、未充值”时产生的奖励，不符合新规则。

### 10.2 上线策略建议

建议二选一：

#### 方案 A：从上线时刻起按新规则生效，不追历史

优点：

- 风险低
- 实施快

缺点：

- 老数据口径不干净
- 用户看到的累计收益可能混有旧逻辑金额

#### 方案 B：上线前做一次历史重算

重算原则：

1. 保留 `inviter_id` 关系。
2. 扫描历史充值成功订单。
3. 按新返现比例重建 `invite_rebate_records`。
4. 重新汇总每个邀请人的：
   - `aff_quota`
   - `aff_history_quota`

优点：

- 数据口径统一

缺点：

- 需要额外编写迁移脚本
- 若历史期间返现比例并非固定值，需人工确认口径

推荐结论：

如果当前系统还未大规模运营，建议采用方案 B。
如果当前系统已运行较久，建议采用方案 A，并在版本说明中声明“新规则自某日上线后生效”。

## 11. 验收标准

### 11.1 业务验收

1. A 邀请 B 注册，B 注册成功：
   - B 获得注册赠送余额
   - A 不获得返现
2. B 成功充值到账后：
   - A 按后台设置比例获得返现
   - A 的 `aff_quota`、`aff_history_quota` 正确增加
3. 同一订单重复回调：
   - A 不能重复获得返现
4. 用户端能看到：
   - 邀请收益统计
   - 已邀请用户列表
   - 建议同时可见返现明细

### 11.2 技术验收

1. SQLite、MySQL、PostgreSQL 三种数据库都可正常迁移。
2. 不删除任何原有邀请字段。
3. 充值主流程不因新增返现逻辑而出现重复加款或事务不一致。
4. OAuth 注册与普通注册行为一致。
5. 管理员补单也能正确触发返现。

## 12. 推荐实施顺序

1. 新增 `InviteTopupRebateRate` 配置项
2. 新增 `invite_rebate_records` 模型与迁移
3. 改造注册逻辑，取消邀请人注册即返现
4. 在各充值成功链路接入统一返现结算
5. 增加用户侧查询接口
6. 改造前端设置页与邀请页
7. 根据选择执行历史数据迁移或保留旧口径

## 13. 最终建议

从兼容原版数据库结构和最小改动成本来看，本次二开的推荐方案是：

1. 复用 `users.inviter_id / aff_count / aff_quota / aff_history`。
2. 复用 `QuotaForInvitee` 作为“邀请注册赠送余额”。
3. 停用 `QuotaForInviter` 的注册返现语义。
4. 新增 `InviteTopupRebateRate` 配置项。
5. 新增 `invite_rebate_records` 表承载返现流水和幂等。
6. 统一在充值成功链路结算邀请返现。

这样能在不破坏原版用户表结构的前提下，完成业务逻辑修正，并支撑后续邀请列表、返现明细、财务追溯能力。
